# WoWLook 魔兽世界装备查询小程序 - 开发文档

## 一、项目概述

### 1.1 产品定位
微信小程序，专用于查询魔兽世界**至暗之夜（12.0）赛季1**大秘境和团本的装备掉落信息。用户通过选择职业、专精、副属性，快速定位目标装备的掉落来源。

### 1.2 核心使用场景
> 玩家是武僧酒仙，需要暴击/全能的装备 → 打开小程序 → 选职业「武僧」→ 选专精「酒仙」→ 勾选「暴击+全能」→ 看到所有暴击全能的装备在哪个副本哪个Boss掉落，包括皮甲部位件、项链、戒指、披风、酒仙可用的武器和饰品，以及是大暴击还是大全能。

### 1.3 版本范围
- 资料片：至暗之夜（Midnight）12.0.1
- 赛季：赛季1（M+ Season 17）
- 装等基准：勇士 1/6（ilvl 246）
- 语言：仅中文

## 二、数据来源

### 2.0 当前首选方案：游戏内插件导出（方案A）

当前项目优先使用 `WoWLookExport` 插件在游戏客户端内直接导出冒险指南数据，再由本地脚本转换为小程序数据文件。

**流程：**

1. 在 WoW 客户端内运行 `/wowlook export`
2. 插件扫描 8 个大秘境和 3 个团本的冒险指南掉落
3. 数据写入 `WTF/Account/<账号>/SavedVariables/WoWLookExport.lua`
4. 本地运行 `node scripts/parse-export.js --input <SavedVariables路径>`
5. 生成 `miniprogram/data/{class}.json/.js`

**优点：**

- 数据直接来自游戏客户端，中文客户端可直接导出中文名称
- 维护方式简单，每赛季或大补丁后手动重新导出一次即可
- 不依赖运行时外部 API 组合来决定副本 / Boss / 物品基础结构

> 下文保留 API 方案，作为插件方案不可用时的回退方案（方案B）。

### 2.1 三源组合方案

#### 当前赛季掉落真值来源（KeystoneLoot + Wowhead 页面）

确定「当前赛季这件装备是否真的入库」，以及它对应的副本和 Boss。

**主要真值源：KeystoneLoot 插件数据（M+副本）**

KeystoneLoot 是一个开源 WoW 插件，其 `data/dungeons.lua` 由插件自动从游戏客户端提取，包含当前赛季每个M+副本的精确物品ID列表。

| 配置项 | 值 |
|--------|-----|
| 仓库地址 | `https://github.com/Wolkenschutz/KeystoneLoot` |
| 掉落表文件 | `data/dungeons.lua` → `lootTable` 字段 |
| Boss归属文件 | `data/items.lua` → 每件物品的 `encounterIndex` 字段 |
| 数据来源 | 游戏客户端自动提取（`KeystoneLoot Updater`） |
| 适用范围 | 8个M+大秘境副本 |

**补充真值源：Wowhead 页面（团本 + 交叉校验）**

| 用途 | 页面类型 |
|------|---------|
| 团本掉落（KeystoneLoot不覆盖） | 对应团本的 `Rewards / Gear / Loot` 页面 |
| 当前赛季总掉落池交叉校验 | `Season 1 Mythic+ Rewards / Loot Drops` |
| 单件装备补充校验 | `item={itemId}` 物品页 |

**使用原则：**

- M+副本掉落以 KeystoneLoot `dungeons.lua` 为主真值（已验证与游戏内一致）
- 团本掉落以 Wowhead 页面为真值
- 一件装备只有在真值源中出现，才允许入库
- Boss 归属优先取 KeystoneLoot `items.lua`；若缺失，用 Wowhead 页面补齐
- 禁止把暴雪 `journal-encounter.items` 直接当成当前赛季掉落表

#### 暴雪 Battle.net API（获取装备基础信息）

| 配置项 | 值 |
|--------|-----|
| OAuth Token端点 | `https://oauth.battle.net/token` |
| API基础地址 | `https://us.api.blizzard.com` |
| 认证方式 | Bearer Token（Header: `Authorization: Bearer {token}`） |
| namespace | `static-us`（物品/副本数据）、`dynamic-us`（赛季数据） |
| locale | `zh_CN` |

**关键接口：**

| 用途 | 接口 |
|------|------|
| 获取OAuth Token | `POST https://oauth.battle.net/token`（client_credentials） |
| 当前赛季大秘境列表 | `GET /data/wow/mythic-keystone/season/{seasonId}?namespace=dynamic-us&locale=zh_CN` |
| 副本详情（Boss列表） | `GET /data/wow/journal-instance/{instanceId}?namespace=static-us&locale=zh_CN` |
| 物品基础信息 | `GET /data/wow/item/{itemId}?namespace=static-us&locale=zh_CN` |

**使用原则：**

- 暴雪 API 只负责稳定元数据：物品 ID、中文名、部位、物品类型、甲种
- `journal-instance` 只用于补副本 / Boss 基础信息，不作为当前赛季掉落真值
- `journal-encounter` 不参与入库判定，避免把历史静态掉落误录为当前赛季掉落

#### Wowhead Tooltip API（获取指定装等的精确属性数值）

```
GET https://nether.wowhead.com/tooltip/item/{itemId}?dataEnv=1&locale=4&ilvl=246
```

| 参数 | 值 | 说明 |
|------|-----|------|
| dataEnv | 1 | 正式服 |
| locale | 4 | 中文（zh_CN） |
| ilvl | 246 | 勇士 1/6 装等 |

返回JSON格式，`tooltip`字段为HTML，从中解析属性数值。

### 2.2 为什么需要三源组合

| 需求 | KeystoneLoot | Wowhead 页面 | 暴雪API | Wowhead Tooltip |
|------|-------------|-------------|---------|-----------------|
| M+当前赛季是否掉落 | ✓ | 交叉校验 | ✗ | ✗ |
| 团本当前赛季是否掉落 | ✗ | ✓ | ✗ | ✗ |
| 掉落来源（副本+Boss） | ✓ | 补充 | 仅基础信息 | ✗ |
| 物品ID、名称、部位、甲种 | 仅ID | 可补充 | ✓ | ✗ |
| 指定装等（246）的精确数值 | ✗ | ✗ | ✗（只返回基础装等） | ✓ |

核心原则：

- M+副本掉落真值以 KeystoneLoot 数据为准（已通过游戏内对照验证一致）
- 团本掉落真值以 Wowhead 页面为准
- 物品结构化字段以暴雪 API 为准
- 246 装等属性数值以 Wowhead Tooltip 为准
- 冲突时以 KeystoneLoot / Wowhead 当前赛季数据为准；无法确认时该物品不入库

#### 筛选逻辑

   - 部位件、项链、戒指、披风不受影响（同职业各专精通用）
   - 武器和饰品过滤为仅该专精可用的

   - 没有常规副属性的饰品/特效装备会被排除（合理行为：按属性筛选时用户不关心纯特效饰品）

| 急速 | haste | 急速 |
| 精通 | mastery | 精通 |
| 全能 | versatility | 全能 |

### 3.4 大属性 / 小属性判定规则

常规护甲、首饰、披风、武器通常有两条副属性；部分饰品和特效装备可能没有常规副属性。

判定方法：从 Wowhead Tooltip 返回的 246 装等数据中，比较两条副属性的数值大小：
- 值大的 → 大属性
- 值小的 → 小属性
- 值相等 → 均标记为「等权」

若装备没有两条常规副属性：

- 不做大小属性判定
- 仍然允许入库
- 由 `stats.secondary = []` 表示

### 4.2 筛选逻辑（小程序运行时）

用户选择职业后加载对应JSON文件，在内存中按专精和属性过滤：

```javascript
// 从对应职业JSON加载数据后，扁平化所有物品并筛选
function filterItems(allItems, { specId, stat1, stat2 }) {
  return allItems.filter(item => {
    // 专精过滤：如果选了专精，只保留该专精可用的装备
    if (specId && !item.specs.includes(specId)) return false;
    // 属性过滤：如果选了属性，只保留包含所选属性的装备
    const itemStats = item.stats.secondary.map(s => s.type);
    if (stat1 && !itemStats.includes(stat1)) return false;
    if (stat2 && !itemStats.includes(stat2)) return false;
    return true;
  });
}
```

## 五、数据拉取脚本设计

### 5.1 脚本职责

Node.js 脚本，负责从 KeystoneLoot 数据、暴雪 API、Wowhead Tooltip 拉取数据，按职业分类后生成13个JSON文件。

**脚本路径：** `scripts/fetch-data.js`

### 5.2 执行流程

```
1. 解析 KeystoneLoot dungeons.lua → 获取8个M+副本的物品ID白名单
       ↓
2. 解析 KeystoneLoot items.lua → 获取每件物品的职业/专精可用性（classes字段）
       ↓
3. 遍历白名单中所有物品ID：
   a. 暴雪 API → 获取基础信息（中文名、部位、甲种）
   b. Wowhead Tooltip → 获取 246 装等的精确属性（副属性、主属性、耐力）
       ↓
4. 通过暴雪 API journal-encounter 确定每件物品的Boss归属
   （仅用于Boss映射，不用于判定是否掉落——掉落以KeystoneLoot白名单为准）
       ↓
5. 按职业分组：根据 items.lua 的 classes 字段，将物品分配到对应职业
   同一物品可能出现在多个职业的文件中（如项链、戒指）
       ↓
6. 生成 data/{className}.json（13个文件）
```

### 5.3 数据过滤与分类规则

**入库规则：**

- 只有出现在 KeystoneLoot `dungeons.lua` 白名单中的物品才允许入库（M+副本）
- 暴雪 API 标记为可装备，且属于护甲、武器、项链、戒指、披风或饰品
- 项链、戒指、披风、饰品、武器统一记为 `armorType: none`
- 没有常规副属性的饰品 / 特效装备仍然入库，不再排除

**按职业分配规则：**

- 根据 KeystoneLoot `items.lua` 的 `classes` 字段判定每件物品可用的职业和专精
- `classes` 结构：`{ [classId]: [specId1, specId2, ...] }`
- 一件物品在某个职业文件中出现时，`specs` 字段只包含该职业下可用的专精ID
- 同一物品可能出现在多个职业文件中（如通用项链、戒指）

**禁止行为：**

- 禁止使用 `journal-encounter.items` 直接生成当前赛季掉落
- 禁止因为旧副本 Journal 数据存在同名副本，就默认其掉落仍与当前赛季一致

### 5.4 Wowhead Tooltip 解析规则

从tooltip HTML中提取副属性：

```javascript
// 解析Wowhead tooltip中的副属性
function parseSecondaryStats(tooltipHtml) {
  const stats = [];
  const statMap = {
    'rtg36': { type: 'haste', name: '急速' },
    'rtg32': { type: 'crit', name: '暴击' },
    'rtg49': { type: 'mastery', name: '精通' },
    'rtg40': { type: 'versatility', name: '全能' }
  };
  for (const [rtg, info] of Object.entries(statMap)) {
    const regex = new RegExp(`<!--${rtg}-->(\\d[\\d,]*)${info.name}`);
    const match = tooltipHtml.match(regex);
    if (match) {
      stats.push({
        type: info.type,
        name: info.name,
        value: parseInt(match[1].replace(',', ''))
      });
    }
  }
  // 标记大小属性
  if (stats.length === 2) {
    if (stats[0].value > stats[1].value) {
      stats[0].isMajor = true;
      stats[1].isMajor = false;
    } else if (stats[0].value < stats[1].value) {
      stats[0].isMajor = false;
      stats[1].isMajor = true;
    } else {
      stats[0].isMajor = true;
      stats[1].isMajor = true;
    }
  }
  return stats;
}
```

### 5.5 请求频率控制

- 暴雪API：每个请求间隔 100ms，避免限流
- Wowhead Tooltip：每个请求间隔 200ms，礼貌拉取
- Wowhead 页面抓取：每个页面请求间隔 300ms，并缓存原始HTML
- 总预估：约 200 件装备，首次全量拉取约 3-5 分钟

## 八、注意事项

1. **数据准确性是核心**：M+副本以 KeystoneLoot 数据为准（已验证与游戏客户端一致），团本以 Wowhead 页面为准，最后通过游戏内抽查验证
2. **旧副本已解决**：执政团之座、艾杰斯亚学院、萨隆矿坑、通天峰的掉落表通过 KeystoneLoot 获取，不再使用暴雪 Journal 静态数据
3. **饰品不再排除**：部分饰品只有特效、没有常规双副属性，也必须入库；此时 `stats.secondary` 允许为空数组
4. **暴雪 Journal 的边界**：`journal-instance` 和 `journal-encounter` 只能当资料参考，不能当当前赛季掉落真值
5. **职业/专精数据来源**：每件物品可用的职业和专精信息来自 KeystoneLoot `items.lua` 的 `classes` 字段，与游戏客户端一致
6. **KeystoneLoot 数据更新**：每次赛季轮换后，拉取最新的 KeystoneLoot `dungeons.lua` 和 `items.lua` 即可获得新赛季的准确掉落表和职业适用性
7. **包大小**：13个职业JSON文件，每个约 10-30KB，按需加载不会有包大小问题
8. **API密钥安全**：暴雪 API 的 Client ID / Secret 仅在数据拉取脚本中使用，不会打包进小程序
