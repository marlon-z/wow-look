# WoWLook 魔兽世界装备查询小程序 - 开发文档

## 一、项目概述

### 1.1 产品定位
微信小程序，专用于查询魔兽世界**至暗之夜（12.0）赛季1**大秘境和团本的装备掉落信息。用户通过选择职业、专精、副属性，快速定位目标装备的掉落来源。

### 1.2 核心使用场景
> 玩家是武僧酒仙，需要暴击/全能的装备 → 打开小程序 → 选职业「武僧」→ 选专精「酒仙」→ 勾选「暴击+全能」→ 看到所有暴击全能的装备在哪个副本哪个Boss掉落，包括皮甲部位件、项链、戒指、披风、酒仙可用的武器和饰品，以及是大暴击还是大全能。

### 1.3 版本范围
- 资料片：至暗之夜（Midnight）12.0.1
- 赛季：赛季1（M+ Season 17）
- 当前版本样本装等：勇士 1/6（ilvl 246）
- 语言：仅中文

## 二、数据来源

### 2.1 当前数据源

当前项目使用 `WoWLookExport3` 插件作为唯一数据源。

所有装备数据都直接来自游戏客户端内的 `冒险者手册`，再由本地脚本转换成小程序使用的数据文件。

### 2.2 标准流程

1. 在 WoW 客户端内运行 `/wowlook3`
2. 插件识别当前赛季地下城与团本
3. 插件逐个副本、逐个首领扫描掉落清单
4. 插件补全每件装备在当前赛季、当前难度下的实际展示版本
5. 插件读取装备说明框，导出原始文本和结构化字段
6. 数据写入 `WTF/Account/<账号>/SavedVariables/WoWLookExport3.lua`
7. 本地运行 `node scripts/parse-export.js --input <SavedVariables路径>`
8. 生成小程序使用的数据文件

### 2.3 数据原则

- 插件导出结果是唯一主数据
- 冒险者手册显示什么，最终数据就必须是什么
- 掉落来源来自插件扫描到的首领掉落关系
- 装备属性来自插件读取到的当前赛季说明框

### 2.4 当前采集链路

当前插件的采集链路如下：

1. 识别当前赛季实例范围
2. 逐个实例、逐个首领读取掉落清单
3. 建立 `副本 -> 首领 -> 装备` 的掉落索引
4. 对每件装备补全当前赛季展示版本
5. 从该展示版本的说明框中读取最终属性
6. 导出原始说明框文本与结构化结果

### 2.5 关键说明

- 首次读取掉落清单，只负责确认掉落来源
- 最终属性不直接取第一次返回的基础结果
- 必须补全当前赛季实际展示版本后，才能读取最终说明框

#### 筛选逻辑

1. **选职业（必选）**：显示该职业所有专精能用的全部装备
   - 对应甲种的部位件（头肩胸腕手腰腿脚）
   - 通用装备（项链、戒指、披风）
   - 该职业所有专精可用的武器和饰品

2. **选专精（可选）**：在职业结果上进一步过滤
   - 部位件、项链、戒指、披风不受影响（同职业各专精通用）
   - 武器和饰品过滤为仅该专精可用的

3. **选副属性（可选，1-2项）**：在当前结果上过滤包含所选属性的装备
   - 没有常规副属性的饰品/特效装备会被排除（合理行为：按属性筛选时用户不关心纯特效饰品）
   - 不选属性时显示全部装备（包括纯特效饰品）

**副属性（4种）：**

| 属性 | type | 显示名 |
|------|------|--------|
| 暴击 | crit | 暴击 |
| 急速 | haste | 急速 |

### 3.4 副属性展示原则

小程序不额外生成“大属性 / 小属性”标签。  
当前策略是直接展示副属性实际数值，由用户自行判断数值高低。

若装备没有常规副属性：

- 仍然允许入库
- `stats.secondary = []`
- 前端按“特效饰品”或无副属性装备处理

- 饰品：部分饰品限定角色定位（坦克/治疗/输出），`specs` 仅包含可用的专精ID

### 4.2 筛选逻辑（小程序运行时）

用户选择职业后加载对应JSON文件，在内存中按专精和属性过滤：

```javascript
// 从对应职业JSON加载数据后，扁平化所有物品并筛选
function filterItems(allItems, { specId, stat1, stat2 }) {
  return allItems.filter(item => {
    // 专精过滤：如果选了专精，只保留该专精可用的装备
    if (specId && !item.specs.includes(specId)) return false;
    // 属性过滤：如果选了属性，只保留包含所选属性的装备
    const itemStats = item.stats.secondary.map(s => s.type);
    if (stat1 && !itemStats.includes(stat1)) return false;
    if (stat2 && !itemStats.includes(stat2)) return false;
    return true;
  });
}
```

## 五、数据转换脚本设计

### 5.1 脚本职责

Node.js 脚本负责读取插件导出的 `SavedVariables` 文件，把原始导出结果转换为小程序使用的数据文件。

当前职责包括：

- 解析插件导出的原始数据
- 提取副本、首领、装备与说明框字段
- 执行本地结构化整理
- 按职业生成小程序数据文件

**脚本路径：** `scripts/parse-export.js`

### 5.2 输入与输出

**输入：**

- `WTF/Account/<账号>/SavedVariables/WoWLookExport3.lua`

**输出：**

- `miniprogram/data/{class}.json`
- 或中间主表文件，供后续分类脚本继续处理

### 5.3 执行流程

```
1. 读取插件导出的 SavedVariables 文件
       ↓
2. 解析 payload，提取 instances 与 items
       ↓
3. 建立装备主表：
   - 装备ID
   - 装备名
   - 副本
   - 首领
   - 部位
   - 图标
   - 物品等级
    - 升级轨道
   - 职业 / 专精归属
   - 白字
    - 主属性 / 耐力 / 副属性
    - 装备特效 / 使用特效
   - 原始说明框文本
       ↓
4. 执行本地分类与字段整理
       ↓
5. 按职业输出小程序数据文件
```

### 5.4 数据过滤与整理规则

**入库规则：**

- 只接收插件正式导出的装备结果
- 装备必须具备明确来源：副本 + 首领
- 装备必须具备有效说明框结果
- 项链、戒指、披风、饰品、武器统一记为 `armorType: none`
- 没有常规副属性的饰品 / 特效装备仍然入库，不再排除

**分类规则：**

- 掉落来源直接来自插件导出结果
- 白字、主属性、副属性、特效文本直接来自插件说明框解析结果
- 职业 / 专精字段直接来自插件导出结果
- 多主属性按数组原样保留，不压缩成单值

**禁止行为：**

- 禁止重新引入外部掉落白名单决定是否入库
- 禁止用外部说明框数据覆盖插件导出结果
- 禁止因为旧副本存在同名装备，就默认它仍等于旧模板数据

### 5.5 结构整理原则

本地脚本只做结构转换，不额外改写插件已经确认的事实结果：

- 不重新计算大属性 / 小属性
- 不压缩多主属性
- 不改写职业 / 专精归属
- 不改写实际物品等级

### 5.6 执行特征

- 不需要请求外部接口
- 不存在网络限流问题
- 主要耗时来自本地文件解析与字段转换

**筛选区（顶部固定）：**

**结果区（下方滚动）：**

**装备卡片内容（每件装备一张卡片）：**

- 部位分组顺序：头、肩、披风、胸、护腕、手、腰、腿、脚、项链、戒指、饰品、武器
- 同部位内按副本顺序排列（先M+副本后团本），同副本内按Boss顺序

## 八、注意事项

1. **数据准确性是核心**：最终结果以插件导出的说明框为准，不再以外部站点或外部接口为准
2. **旧副本问题已解决**：旧副本回归赛季后，必须先补全当前赛季展示版本，再读取说明框；不能直接使用基础装备模板
3. **饰品不再排除**：部分饰品只有特效、没有常规双副属性，也必须入库；此时 `stats.secondary` 允许为空数组
4. **掉落来源与展示属性必须分层处理**：掉落清单负责来源索引，说明框负责最终属性，二者不能混用
5. **职业 / 专精归属以插件导出为准**：本地脚本只做结构转换，不再重新猜测
6. **赛季更新维护方式已简化**：每次赛季轮换后，重新运行插件导出并重新转换即可
7. **包大小**：13个职业JSON文件，每个约 10-30KB，按需加载不会有包大小问题
