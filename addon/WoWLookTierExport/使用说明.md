# WoWLookTierExport 使用说明

这个插件用于导出当前赛季职业套装数据，输出内容包括：

- 13 个职业的 117 件套装外观装备
- 每个职业 5 件带 `2 件 / 4 件` 效果的核心套装
- 每个职业 4 件配套外观散件（披风、护腕、腰带、鞋子）
- 每件套装的英雄轨道展示 tooltip
- 套装名、部位、护甲类型、主副属性
- 按专精展开的 `2 件 / 4 件` 套装效果

## 命令

```text
/wowtierexport all
/wowtierexport monk
/wowtierexport monk,druid,mage
/wowtierexport summary
/wowtierexport help
```

## 导出结果

导出后会写入：

```text
WTF/Account/<账号>/SavedVariables/WoWLookTierExport.lua
```

核心字段：

- `summary`
- `classes`
- `payload`

其中：

- `classes` 是 Lua 表，适合直接调试查看
- `payload` 是 JSON 字符串，适合后续 Node 脚本解析

## 数据边界

这版插件当前使用统一的英雄赛季展示 link：

- 难度：英雄
- 轨道：英雄 `2/6`

所以导出的是真实可渲染的英雄展示版本，不是玩家个人升级后的实例值。

装备 ID 的边界也要分清：

- `bonusItemIds`：5 件核心套装，负责导出装备数据和专精套装效果。
- `appearanceItemIds`：4 件配套外观散件，只导出装备数据，不读取专精套装效果。

两组 ID 都是 Wowhead / 游戏资料中确认过的真实 `itemId`。插件不会再运行时从幻化 `source` 自动猜 4 件散件，因为同一外观槽位可能返回多个 source，曾经导致 DK 鞋、DZ 鞋、SS 腰带等少数槽位选错物品。

## 数据来源

这个插件的原则是：

**只手工维护稳定的入口 ID，其他展示数据尽量从游戏内 tooltip 读取。**

### 手工维护的数据

手工维护的数据都在 `WoWLookTierExport.lua` 顶部：

```lua
local SEASON_BONUS_ID = 3524
local RAID_HEROIC_DIFFICULTY = 5
local RAID_HEROIC_CONTEXT = 3606

local TIER_SETS = {
    deathknight = {
        setName = "Relentless Rider's Lament",
        bonusItemIds = { ... },
        appearanceItemIds = { ... },
        specs = { ... },
    },
}
```

需要手工维护的只有这些：

- `SEASON_BONUS_ID`：当前赛季/来源需要的 bonus 参数。
- `RAID_HEROIC_DIFFICULTY`：当前用于展示的难度，当前固定英雄。
- `RAID_HEROIC_CONTEXT`：当前英雄团本来源 context。
- `TIER_SETS[*].setName`：英文套装名，主要用于人工核对和兜底展示。
- `TIER_SETS[*].bonusItemIds`：每个职业 5 件有套装效果的装备 ID。
- `TIER_SETS[*].appearanceItemIds`：每个职业 4 件只补外观的散件 ID。
- `TIER_SETS[*].specs`：职业专精列表；只有暴雪新增、删除或调整专精 ID 时才需要改。

### 自动读取的数据

下面这些不要手写，也不要从 Wowhead 页面复制进代码：

- 中文装备名
- 装等
- 轨道文字，例如 `英雄 2/6`
- 部位
- 护甲类型
- 主属性、副属性、耐力、护甲
- 5 件核心套装的中文套装名
- 5 件核心套装的套装成员列表
- 每个专精的 `2 件 / 4 件` 套装效果

这些数据由插件在游戏内通过 `C_TooltipInfo.GetHyperlink()` 读取 tooltip，再解析后写入 `payload`。

## 下版本维护流程

下个赛季更新时，按这个顺序做。

### 1. 找 13 个职业的新套装 ID

数据源优先级：

1. Wowhead 当前版本职业套装页面 / item 页面。
2. 游戏内外观收藏的“套装”界面，用于核对 9 件外观名称。
3. 游戏内聊天框贴出的物品链接，用于确认最终 `itemId`。

每个职业需要维护 9 个 `itemId`：

- 5 个 `bonusItemIds`：头、肩、胸、手、腿。
- 4 个 `appearanceItemIds`：披风、护腕、腰带、鞋子。

注意：

- `bonusItemIds` 必须是 tooltip 里有职业套装 `2 件 / 4 件` 效果的 5 件。
- `appearanceItemIds` 只是同一外观套装的补齐件，不一定有 `setId`，也不应该读取专精套装效果。
- 只填装备 `itemId`，不要填幻化 `sourceId`、`visualId`、appearance ID。

### 2. 核对 4 件外观散件

这一步很重要。之前出错就是因为插件从幻化 source 自动推断 4 件散件，结果 DK 鞋、盗贼鞋、术士腰带、萨满腰带选到了同外观或同槽位下的错误物品。

正确做法：

1. 打开游戏内外观收藏的“套装”界面。
2. 选中对应职业和当前赛季套装。
3. 鼠标移到 9 个部位图标上，记录游戏显示的装备中文名。
4. 用装备名去 Wowhead / 游戏物品链接核对真实 `itemId`。
5. 把披风、护腕、腰带、鞋子的真实 `itemId` 填入 `appearanceItemIds`。

不要再用 `C_TransmogSets.GetSourcesForSlot()` 的返回结果自动挑散件。它返回的是幻化 source 集合，不等于我们要的装备清单，存在同外观多来源、随机属性物品、职业不匹配物品混入的问题。

### 3. 更新赛季 link 参数

如果新赛季的展示 link 参数变化，需要更新：

- `SEASON_BONUS_ID`
- `RAID_HEROIC_DIFFICULTY`
- `RAID_HEROIC_CONTEXT`

当前插件会用这些参数构造类似下面的英雄展示链接：

```text
item:<itemId>::::::::90:0::5:1:<SEASON_BONUS_ID>:1:28:<RAID_HEROIC_CONTEXT>:::::
```

目标不是读取玩家身上实际装备，而是生成一个稳定的“英雄 2/6”展示版本，方便小程序统一展示。

### 4. 导出并验证

进游戏后执行：

```text
/wowtierexport all
```

然后检查 SavedVariables：

```text
WTF/Account/<账号>/SavedVariables/WoWLookTierExport.lua
```

重点看：

- `summary.classCount` 应该是 `13`。
- `summary.itemCount` 应该是 `117`。
- `summary.bonusItemCount` 应该是 `65`。
- `summary.extraAppearanceItemCount` 应该是 `52`。
- 每个职业 `itemCount` 应该是 `9`。
- 每个职业 `bonusItemCount` 应该是 `5`。
- 每个职业 `extraAppearanceItemCount` 应该是 `4`。
- 4 件外观散件的名称和游戏外观套装界面显示一致。

导出后再用脚本转成小程序数据。如果只发现少数散件错了，可以只替换对应职业 `data-4.2.x` 里的那几个散件，不需要重建整份数据。

## 复用说明

这个插件可以复用到下一赛季，但要区分：

- 可复用的是导出框架
- 需要替换的是赛季配置

### 可复用的部分

- 构造赛季展示 link
- 读取 tooltip
- 解析装等、部位、护甲类型、主副属性
- 解析套装名、套装成员
- 按专精导出 `2 件 / 4 件` 效果
- 导出 `summary / classes / payload`

### 下个赛季需要更新的部分

1. `TIER_SETS`
- 13 职业新的 `bonusItemIds`（5 件核心套装）
- 13 职业新的 `appearanceItemIds`（4 件配套外观散件）
- 新的套装名

2. 赛季 link 参数
- `SEASON_BONUS_ID`
- `RAID_HEROIC_DIFFICULTY`
- `RAID_HEROIC_CONTEXT`

也就是说，这个插件不是“永久零维护通用插件”，而是：

**一个可以跨赛季复用的套装导出框架。**

换赛季时通常只需要更新：

- 一份职业套装配置
- 一组赛季展示参数

主逻辑不需要重写。

维护 `TIER_SETS` 时不要填幻化 `sourceId`、`visualId` 或外观来源 ID，只填装备 `itemId`。装备名称、属性、装等、套装效果仍然由游戏内 `C_TooltipInfo.GetHyperlink()` 读取。
